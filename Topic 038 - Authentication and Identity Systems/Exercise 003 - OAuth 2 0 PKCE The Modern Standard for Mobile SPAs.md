# Exercise 003 - OAuth 2.0: PKCE (Mobile & SPA Security)

### Objective
No secrets for the frontend. Master **PKCE (Proof Key for Code Exchange)**— the modern, secure way to do OAuth 2.0 without using a "Secret Key." Learn why the old `Implicit Flow` is now deprecated and understand how PKCE protects mobile apps and Single Page Applications (React/Vue/Angular) from code-interception attacks.

### Core Concepts to Master
- **PKCE (pronounced 'pixie'):** Used by "Public Clients" that cannot safely store a password or a secret key.
- **Code Verifier:** A long, random string generated by the app.
- **Code Challenge:** A "Hash" of the Code Verifier (usually using SHA-256).
- **The Protocol Match:** Google stores the `Challenge` during step 1 and only gives the `token` if the app sends the original `Verifier` in step 2.
- **Implicit Flow (Deprecated):** The old method of sending the `token` directly in the URL (highly insecure).

### The Practical Challenge

**Part 1: The Code-Interception Attack**
1. You have a mobile app. 
2. Someone builds a "Fake app" that registers the same "Redirect URL" as your app.
3. The real app sends the user to Login. 
4. After login, Google sends the `code` back to the phone.
5. The **Fake app** intercepts the `code`.
6. Without a `client_secret` or `PKCE`, the fake app can exchange the `code` for a token and steal the account!
7. Discuss: How does **PKCE** stop this? (Answer: The fake app doesn't know the unique "Code Verifier" generated by the real app).

**Part 2: The Hash Step**
1. Generate a random string: `my_secret_verifier`.
2. Generate its SHA-256 hash (the Challenge): `xyz-challenge`.
3. Put the `Challenge` in the URL for step 1 of the login.
4. Discuss: If an attacker sees your URL, do they know your original `Verifier`? (Answer: No, hashing is a "One-Way" trip).

**Part 3: The Exchange**
1. In step 2, send the `code` + the plain `my_secret_verifier` to Google.
2. Discuss why the Google Authorization server won't give the token to anyone else. 
3. (Answer: Only the app that generated the original Challenge has the matching Verifier).

**Part 4: Identifying the flow**
1. Research the **Auth0** or **Firebase** documentation. 
2. Which flow do they recommend for a React Native app? (Answer: PKCE).

### Common Gotchas & Troubleshooting
- **Gotcha 1:** SHA-256 vs Plain. Some older APIs allowed "Plain" PKCE, which doesn't use hashing. This is useless! Always use `S256` method.
- **Gotcha 2:** Implementing it manually. It is very hard to write a secure PKCE implementation from scratch (handling the cryptographically random strings). Always use a battle-tested library like `AppAuth`.

### Reflection Question
Why is PKCE becoming the "Universal" flow for almost all OAuth 2.0 apps, even backend ones? (Answer: Because it is technically more secure than even the Auth Code Flow with a secret).
